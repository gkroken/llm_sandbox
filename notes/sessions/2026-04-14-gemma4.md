# 2026-04-14 — Gemma 4 Evaluation

**Goal:** Evaluate Google's Gemma 4 models for agentic programming with Continue.dev.

## Background

Gemma 4 released April 2, 2026 under Apache 2.0. Built from Gemini 3 research — a significant
generational leap over Gemma 3. Four sizes: E2B, E4B (edge), 26B MoE (A4B), 31B Dense.

**Key benchmarks vs Gemma 3:**

| Benchmark | Gemma 3 27B | Gemma 4 E4B | Gemma 4 31B |
|---|---|---|---|
| AIME 2026 (math) | 20.8% | 42.5% | 89.2% |
| LiveCodeBench | 29.1% | — | 80.0% |
| τ2-bench (agentic tool use) | 6.6% | — | 86.4% |

All models have native function calling, reasoning/thinking mode, and multimodal support.

## What We Tried

### Adding Gemma 4 E4B to the stack

Added `gemma4` to `switch-model.sh`:
```bash
"gemma4:google/gemma-4-E4B-it:"
```

**First attempt with `vllm/vllm-openai:latest`:** Crashed immediately.
```
ValueError: The checkpoint you are trying to load has model type `gemma4` but
Transformers does not recognize this architecture.
```
Fix: Use the dedicated `vllm/vllm-openai:gemma4` image (requires transformers>=5.5.0).

**After switching image:** Loaded successfully. But speed was ~0.1 tokens/s — completely unusable.

Tried E2B as well — still too slow.

### Tool calling setup (for agent mode)
Gemma 4 requires additional vLLM flags:
```yaml
--enable-auto-tool-choice
--tool-call-parser gemma4
--reasoning-parser gemma4
```

Without these, Continue's Agent mode returns:
```
"auto" tool choice requires --enable-auto-tool-choice and --tool-call-parser to be set
```

## Why It's Slow on the 3070 Ti

Gemma 4 uses heterogeneous head dimensions, which forces vLLM to use the **Triton attention backend**
instead of FlashAttention. Triton is significantly slower on consumer GPUs.

Additionally, no quantized versions of Gemma 4 exist yet (no AWQ/GGUF from Google) — running raw FP16.
Both of these are hard constraints, not config issues.

## What We Concluded

- The 3070 Ti is not the right hardware for Gemma 4
- E4B and E2B are edge models — designed for phones and laptops with NPUs, not CUDA GPUs without quantization
- The impressive benchmarks are for the 26B and 31B models, not E4B
- Isolated Gemma 4 work to `gemma-4-testing` branch for others to evaluate

## Gemma 4 on Production Box (Future)

Recommended model: `google/gemma-4-26B-A4B-it` (MoE — only 4B parameters active during inference, fast)

```yaml
# docker-compose.yml vLLM command for production
--model google/gemma-4-26B-A4B-it
--enable-auto-tool-choice
--tool-call-parser gemma4
--reasoning-parser gemma4
```

Use `vllm/vllm-openai:gemma4` Docker image, not `latest`.

## What We Learned

- vLLM ships dedicated images for new model families — check for `:gemma4`, `:latest` won't work
- Triton attention backend is a hard requirement for Gemma 4 due to architecture — not configurable
- "Edge" models (E2B/E4B) are optimized for mobile NPUs, not CUDA consumer GPUs
- No quantized Gemma 4 exists yet — running FP16 on 8GB is not viable
- Production box (DGX Spark, 128GB) is the right place to properly evaluate Gemma 4
